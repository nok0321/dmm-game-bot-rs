# 01. 目的とスコープ

## 1.1 目的

DMM/FANZA ブラウザゲーム (Canvas/WebGL) の定型操作 (AP 回復 → 討伐周回) を
自動実行する Windows 用 CLI ツールを Rust で実装する。
最終形は **単一の exe バイナリ** として配布可能であること。

主対象タイトル:
- 「あやかしランブル」 (現行 `config/default.toml` のデフォルト)
- 「天地狂乱」 (テンプレ切り出し元、実機 2 サイクル継続成立済み)

## 1.2 自動化対象シーケンス (順序主導 9 ステップ + 霊晶石ガード)

画面アンカーによる状態判定は **行わない**。指定順にクリック対象テンプレを
探して押すだけのシンプルな順序駆動。

実装の `Step` enum は **10 要素**: クリックを発行する 9 ステップに加え、
Step 2.5 (`ReissekiGuard`) が ROI 限定のアサートのみ (クリック発行なし) を担当する。

| Step | enum | クリック対象 | 検出テンプレート |
|------|---|------|------------------|
| 1 | `ApPlus` | AP+ アイコン | `ap_plus_button` |
| 2 | `UseMax` | 「最大選択」 | `ap_recovered_use_max` |
| 2.5 | `ReissekiGuard` | (クリックなし) 霊晶石スロットの選択数=0 を ROI 限定確認 | `reisseki_zero_guard` |
| 3 | `UseButton` | 「使用」 | `use_button` |
| 4 | `TapIndicator` | 回復モーダルの「TAP」 | `tap_indicator` |
| 5 | `Toubatsu` | 「討伐」(朱色) | `toubatsu_button` |
| 6 | `ToubatsuStart` | 「討伐開始!」 (以後ハード sleep ≒ 25 分) | `toubatsu_start` |
| 7 | `Next1` | 周回完了後の「次へ」(1 回目、最大 45 分待ち) | `next_button` |
| 8 | `Next2` | 「次へ」(2 回目) | `next_button` |
| 9 | `Close` | (Optional) 報酬達成モーダルの「閉じる」 | `close_button` |

- ステップ 9 は報酬上限到達後は表示されないため、未検出でも正常パス
  (`StepLog.skipped = true`)。
- 1 サイクル完了後は再びステップ 1 を待ち、`ap_plus_button` の再検出が
  「次サイクル開始」のシグナルになる。

## 1.3 スコープ外

- DMM Games Player (デスクトップアプリ) 経由のゲーム自動化
- 戦闘中の高度な判断 (コマンド選択、対象選択など)
- ログイン処理、2 段階認証
- マルチアカウント並列実行
- 他のブラウザ (Edge, Firefox) のサポート
- macOS / Linux のサポート

## 1.4 非機能要件

| 項目 | 目標値 | 備考 |
|------|--------|------|
| 1 周回あたり CPU 時間 | < 5% (Ryzen / Core i5 相当) | テンプレートマッチングは imageproc が rayon で並列化 |
| メモリフットプリント | < 100MB | テンプレートと直近スクショのみ保持 |
| 通常ステップのポーリング間隔 | 1.5 秒 (タイムアウト 60 秒) | Step 1〜5, 8, 9 |
| 周回後ステップのポーリング間隔 | 7 秒 (タイムアウト 45 分) | Step 7 (Next1) |
| ToubatsuStart 後のハード待機 | 25 分 (コード既定) / 27.5 分 (`default.toml` 既定) | 本物の戦闘 30 分中は一切画面を見ない |
| Stability poll 間隔 | 50 ms (`stability_poll_ms`) | pending stability 中だけ高速ポーリング |
| 配布形態 | 単一 exe (≒ 6.2 MB) | `crt-static` + `lto` + `strip` |
| ウィンドウ位置 | 可変対応 | 各ステップで `client_rect()` を再取得 |
| DPI スケーリング | 100% / 125% / 150% 対応 | `DPI_AWARENESS_CONTEXT_PER_MONITOR_AWARE_V2` |

## 1.5 規約上の注意事項 (設計上の前提)

DMM GAMES 利用規約は通常「ツール、bot、自動化ソフトウェアによるアクセス」を
禁止しています。本ツールの開発・使用にあたっては以下を前提とします:

- 検証用アカウントでの動作確認のみを推奨し、本番アカウントでの常用は利用者の自己責任
- 配布する場合も「教育目的・学習用途」の明示
- ヒューマンライク化 (クリック間隔のジッタ、座標微小ランダム化) は規約回避目的ではなく
  安定動作のため

## 1.6 設計原則

1. **画面に映っているものだけを根拠にする**:
   ゲーム内部状態への侵入はせず、人間がプレイする際と同じ情報源 (画面ピクセル) のみで判断する。
2. **順序主導で進行管理**:
   10 要素 (`Step` enum) の固定シーケンスとして実装。9 要素は指定テンプレートを
   タイムアウト付きで待ち受け→クリックし、Step 2.5 (`ReissekiGuard`) のみ
   ROI 限定アサート (クリック発行なし)。画面アンカー判定はしない。
3. **失敗時は安全側に倒す**:
   想定外の画面が出たら停止 → 人間に通知。リソース消費を伴う操作の取りこぼし対策。
4. **ドライランモード必須**:
   クリック送信を行わず、検出ログのみ出すモードを既定にする (`safety.dry_run = true`)。
5. **設定外部化**:
   テンプレート画像、許容スコア閾値、待機時間などは TOML で外部化。
6. **絶対不変条件は機械的に強制**:
   霊晶石ガードの threshold 下限 (0.80) と ROI 必須は `Config::validate` で起動時に弾く。
